# Clean Architecture — Quick Reference

## Import Rules Decision Matrix

| Writing code in... | Can import from...                                  |
|--------------------|-----------------------------------------------------|
| `domain/`          | stdlib only                                         |
| `port/`            | domain/                                             |
| `app/`             | domain/, port/, config/                             |
| `adapter/`         | app/, domain/, port/, config/, pkg/, third-party    |
| `bootstrap/`       | all layers + third-party                            |
| `cmd/`             | bootstrap/ only                                     |
| `config/`          | stdlib, viper, envconfig (no business packages)     |
| `pkg/`             | stdlib only                                         |

## "Where does this code live?" Flowchart

```
Is it a business entity or value object?
  → domain/

Is it an interface (contract)?
  → port/  (repo, service, event publisher, cache)

Is it a use case that orchestrates domain + port?
  → app/service/

Is it a concrete implementation (DB, HTTP, Kafka)?
  → adapter/  (db/, http/, kafka/, redis/)

Is it wiring/DI?
  → bootstrap/

Is it main()?
  → cmd/<service>/main.go

Is it a config struct?
  → config/

Is it a reusable utility with no business logic?
  → pkg/  (logger, httputil, validator, etc.)
```

## Common Mistakes

| Mistake                                         | Fix                                              |
|-------------------------------------------------|--------------------------------------------------|
| domain imports a DB driver                      | Move DB types to adapter, use interfaces in port |
| app imports gin.Context                         | Extract payload in adapter, pass plain struct    |
| adapter imports another adapter                 | Extract shared logic to pkg/ or domain/          |
| bootstrap logic in cmd/main.go                  | Move wiring to bootstrap/, keep main tiny        |
| config struct has DB query methods              | Config is data only — queries belong in adapter  |

## Layer Size Heuristics

- `domain/` entity — 20-60 lines
- `port/` interface — 5-15 lines (keep small, ISP)
- `app/service/` — 50-150 lines per service
- `adapter/db/` query — generated by sqlc, hand-wrap in repo
- `bootstrap/` — one big `Wire()` / `Setup()` function per service

## Checklist Before Writing a New File

- [ ] What layer does this belong to?
- [ ] Does my import list violate the matrix above?
- [ ] Am I depending on a concrete type that should be an interface?
- [ ] Is this file doing more than one thing (SRP)?
- [ ] Will this be unit-testable without spinning up real infrastructure?

## Interface Naming Convention

```go
// port/ — noun describing capability
type PatientRepository interface { ... }
type NotificationSender interface { ... }
type CacheStore interface { ... }

// app/ — service name matches use case
type PatientService struct { repo port.PatientRepository }
```
